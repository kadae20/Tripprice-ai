# Recommender — 추천 엔진 소울 파일

## 왜 이 파일이 존재하는가

추천의 질이 곧 데이터 독점의 속도다.
좋은 추천 → 사용자가 선택 → 데이터 축적 → 더 좋은 추천.
이 선순환이 멈추면 독점도 멈춘다.

## 모델 선택 기준

| 상황 | 모델 | 이유 |
|------|------|------|
| 일반 추천 (도시 + 목적 + 예산) | `claude-haiku-4-5-20251001` | 빠름, 저렴, 충분한 품질 |
| 복잡한 조건 (다중 우선순위, 특수 요청) | `claude-sonnet-4-6` | 정확도 우선 |
| 취향 분석, 장기 패턴 파악 | `claude-opus-4-6` | 느리지만 고품질 |
| Extended Thinking | 취향 분석에만 | 일반 추천에 사용 금지 (응답 지연) |

**기본값은 Haiku.** Sonnet/Opus는 명시적 근거가 있을 때만.

## 응답 원칙

### 반드시 포함할 것

```json
{
  "fit": ["이유1", "이유2", "이유3"],   // 3개 필수
  "caution": "솔직한 단점 1개",          // 없으면 신뢰도 하락
  "one_line": "15자 이내 핵심 한 줄"
}
```

- **`fit` 3개 필수** — AI가 누락하면 서버에서 폴백 자동 생성. 하지만 AI가 채워야 가치 있다.
- **`caution` 1개** — 감언이설 금지. 솔직한 단점이 신뢰를 만든다.
  - 나쁜 예: "완벽한 호텔입니다" → caution 없음
  - 좋은 예: "엘리베이터가 느려 고층 이동 불편할 수 있음"
- **감언이설 금지** — "최고의", "완벽한", "놓치면 안 될" 표현 지양

### JSON Structured Output 강제

프롬프트에 항상 스키마 명시:
```
반드시 아래 JSON 형식으로만 응답하라. 마크다운 코드블록 없이 순수 JSON:
{ "hotels": [ { "hotel_id": "...", "fit": [...], "caution": "...", "one_line": "..." } ] }
```

## 검증 흐름

1. AI 응답 수신
2. `fit.length >= 1` 확인 → 없으면 1회 재요청
3. 재요청도 실패 → 서버 폴백값 자동 주입
4. 최종 응답은 항상 `fit` 3개 보장

## 추천 품질 기준

좋은 추천이란:
- 사용자가 선택한 후 "역시 AI가 맞혔다"고 느끼는 것
- caution을 보고 "이 서비스는 솔직하다"고 신뢰하는 것
- 다음 여행에도 여기서 시작하게 만드는 것
